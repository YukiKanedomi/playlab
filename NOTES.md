# Playlab — 構想メモ（Yorozuyaからの引き継ぎ）

> 「思いつきのゲームをどんどん形にして公開・試せる実験場」。1リポジトリ＝モノレポで複数作品。叩き台であり確定ではない。原則は CLAUDE.md に要約済み。

## いちばん大事なこと：楽しく遊ぶ・遊び心
- **何よりまず「楽しい」か**。技術や構成は手段で、目的は**遊んで楽しいこと**。判断に迷ったら「で、これ遊んで楽しい？」に立ち返る。
- **遊び心を全体に**：作品そのものはもちろん、ハブ・演出・ちょっとしたインタラクション・隠し要素まで、**作っていて楽しい/触って楽しい**を優先。
- 完璧さより**まず動かして遊ぶ**。荒くても面白い芽を早く触れる形に（思いつき→即試遊）。
- 手触り（juice）・気持ちよさは「楽しい」の核なので大切にする。

## コンセプト
- **名前**：Playlab（遊びの実験室）。repo候補 `playlab`。
- **思いつきを即・形に**：ゲームのアイデアを最短で動く形にして**公開して試せる**。
- **1リポジトリに複数作品（モノレポ）**：作品ごとにリポジトリを切らない。手間と管理を最小化。
- **ノウハウの使いまわし**：1つのプロジェクト内で作るので、入力・音・セーブ・演出などを**即流用**できる。
- **他プロジェクトへの波及**：ゲーム作りで磨いた技術（Canvas描画・状態管理・パフォーマンス・アニメ/演出・タッチ操作）は**他PJにも持ち出せる**資産にする。
- スマホ前提（リモート運用）：**タッチ操作・レスポンシブ**を最初から標準に。

## 最重要原則：ノウハウは共有、発想は自由
- ノウハウ（`shared/`）は**使い回せるようにする。ただし各作品はそれに決して縛られない**。
- `shared/` は「**使ってもいい道具箱（opt-in）**」であって「従うべき枠（framework）」ではない。**shared を一切使わずに1本作ってもOK**。
- 各ゲームは**完全独立**：独自のフォルダ・独自の依存・独自の構造/作風で自由に作れる。共通の作法は「雛形のコピー元」程度の緩い出発点にとどめ、強制しない。
- 設計判断のときは常にこの原則を優先：**「共通化のために自由を削る」ことはしない**。便利だから使う、合わなければ捨てる。
- 結果として `shared/` は「薄く・任意・後付けで育つ」。共通化は“発見されたら切り出す”（先に枠を作って嵌めない）。

## 技術方針（決定）
- **Vite + 作品ごと自由**：共通の `shared/` utils は持ちつつ、各ゲームは必要に応じて **Phaser / three.js など好きなライブラリ**を使える。表現の幅を最大化。
  - トレードオフ：共通化のうまみはやや減る → `shared/` は「薄い共通基盤＋規約」に留め、重い部分は各ゲーム任意。
- **公開は GitHub Pages**（全体方針どおり）。`base: '/playlab/'`、データ取得は `import.meta.env.BASE_URL` 起点。
- **マルチページビルド**：1回の push でハブ＋全ゲームをまとめて公開。

## アーキテクチャ（叩き台）
```
playlab/
├─ index.html          ハブ：全作品をカード一覧→クリックで起動
├─ games.json          作品レジストリ（title/desc/thumb/tags/date/path）データ駆動
├─ shared/             ノウハウの貯金箱（薄い共通基盤・任意）
│   ├─ loop.ts         ゲームループ/時間管理（requestAnimationFrame, fixed-step）
│   ├─ input.ts        キー/タッチ/ゲームパッド（スマホ対応の肝）
│   ├─ audio.ts        SE/BGM
│   ├─ save.ts         localStorage セーブ
│   ├─ juice.ts        画面揺れ/パーティクル/トゥイーン（“気持ちよさ”）
│   └─ math/rng.ts     ベクトル/衝突/乱数
├─ games/
│   ├─ <作品slug>/index.html + main.ts（＋任意で phaser/three.js）
│   └─ ...
└─ vite.config.ts      base:'/playlab/'、rollup multi-page input
```

## 公開・運用フロー
- 新作：`games/<slug>/` を足す → `games.json` に1行追記 → push で公開。
- ハブはデータ駆動（games.json）なので、カードは自動で増える。サムネ・タグ・日付で整理。
- 各ゲームは独立して起動・試遊できる（URLで直リンク共有も可）。

## ノウハウ蓄積の設計（※ただし任意・非強制）
- `shared/` に「入力・音・セーブ・juice・数学」を集約 → 2作目以降は**使いたければ即流用**（必須ではない）。
- “juice”（手触り・演出）を共通部品化しておくと、どの作品もすぐ気持ちよくできる。
- 共通の**雛形（テンプレ）**は1つ用意するが、あくまで**コピー元の出発点**。中身は自由に捨てて書き換えてよい。
- 共通化は**「同じものを2〜3回書いたら切り出す」**くらいの後追いで。先に抽象化して各作品を縛らない。

## 作り方は自由（アプローチを定めない）
- **原則：どんなアプローチでもOK**。決まったモードや手順は設けない。発想の自由が最優先。
- 例（あくまで一例、これに限らない）：完全オリジナル／既存ゲームの模写で「作りと面白さ」を学ぶ習作／ジャンル横断の実験／既存の改変・組み合わせ／ネタ一発 など、何でもアリ。
- どのアプローチでも、得た知見は**使いたければ** `shared/` やスキルに抽出してよい（任意）。
- **権利の配慮（嘘をつかない/誠実）**：既存作を参考にする場合は学習目的に留め、**商標名・公式アート/音源/ロゴはコピーしない**。**仕組みを参考にしつつ、絵・音・名前は自作/フリー素材**に。元ネタは「〜に学ぶ習作」等とクレジット。公開時は特に注意。

## スキル化（横断再利用）— 積極的に進める
- **二層の再利用**：`shared/`＝Playlab内のコード再利用、**スキル＝全PJ横断のワークフロー/知識再利用**。汎用的に役立つものは**積極的にスキル化**して `~/.claude/skills/` に置く（＝他プロジェクトでも自動で使える＝技術波及の本命）。
- スキル候補（例）：
  - **game-scaffold**：雛形から新ゲームを生成（games.json 追記まで）。
  - **game-juice**：画面揺れ/パーティクル/トゥイーン等の“手触り”を後付けする定石集。
  - **web-game-deploy**：Vite マルチページ＋GitHub Pages（base/BASE_URL）公開の定型。
  - **game-clone-study**：既存ゲームを分解→コアループ抽出→習作として再現する手順（権利配慮込み）。
  - **sprite/asset 準備**：フリー素材取得（web-free-images 連携）・スプライト整理。
  - **game-playtest**：render-check/webapp-testing でモバイル/PC両幅の動作・操作確認。
- 作り方：`skill-creator` スキルで作成。「同じ手順を2〜3回やった」と気づいたらスキルに昇格（共通化と同じ後追いの発想）。

## デザイン（共通の最重要方針）
- **“AIっぽい没個性”を避ける**：テンプレ然とした既定デザイン（絵文字の多用、ありがちなグラデ/カード/紫系、量産フォント、意味のない装飾）にしない。**意図のある・個性のあるデザイン**を狙う。
- **世の中の良い実例をふんだんに参考にする**：受賞インディーのサイト/UI、名作ゲームのタイトル・HUD・メニュー、洒落たランディングページ等を**具体的に集めて引く**（雰囲気だけでなく、なぜ良いかを言語化して取り込む）。
- 作風の自由は保ちつつ、**質の下限は高く**：各ゲームの世界観は自由だが「安っぽい/没個性」は避ける。ハブは特に洗練・シンプルに。
- ツール：`frontend-design`（方向性）/ `theme-factory`（配色・フォント）/ `render-check`（目視QA）/ `web-free-images`（実素材）。
- 具体UI（ハブ）：サムネ＋タイトル＋短い説明＋タグのカード、新着順/タグ絞り込み。配色・ロゴは新規検討（必要ならワードマーク案）。

## 新プロジェクトでの着手の一歩
- 小さく：**ハブ + games.json + shared 薄い基盤 + サンプルゲーム1本（例: スネーク or ブロック崩し）** をscaffold。
- Vite マルチページ＋GitHub Pages（base/BASE_URL）を最初に通して「push→公開→スマホで試遊」を確立。
- `shared/input.ts` のタッチ対応を最優先（スマホ運用のため）。
- 雛形フォルダ（new-game-template）を用意して、以後はコピーで量産。

## 参考・連携
- 公開は GitHub Pages 既定（全体方針）。可視化やハブ的なものは viz-panels も参考。
- 他PJへ持ち出す技術資産：Canvas描画・状態管理・ゲームループ・タッチ入力・演出（juice）・パフォーマンス最適化。
